# Домашнее задание 01: Документирование архитектуры в Structurizr

## Вариант 10. Планирование задач

Проектируемая система — сервис планирования задач, похожий по назначению на LeaderTask. Система помогает пользователям формулировать цели, создавать задачи на пути к этим целям, назначать исполнителей и отслеживать статус выполнения задач.

## Исходные данные варианта

Приложение должно содержать следующие данные:

- цель;
- задача;
- исполнитель.

Необходимо реализовать API:

- создание нового пользователя;
- поиск пользователя по логину;
- поиск пользователя по маске имени и фамилии;
- создание новой цели;
- получение списка всех целей;
- создание новой задачи на пути к цели;
- получение всех задач цели;
- изменение статуса задачи в цели.

## Архитектурный подход

Архитектура описана сверху вниз: сначала выделены пользователи и внешние системы, затем сама система, после этого — контейнеры и их взаимодействие. Такой подход подходит для первого архитектурного описания, потому что позволяет сначала показать систему в контексте, а уже потом перейти к внутреннему устройству.

Для варианта выбрана простая service-based / layered архитектура:

- отдельное веб-приложение отвечает за пользовательский интерфейс;
- API Gateway принимает HTTP-запросы от клиента;
- доменные сервисы разделены по основным объектам предметной области: пользователи, цели, задачи;
- PostgreSQL хранит пользователей, цели, задачи и историю изменений статусов;
- Notification Service отвечает за отправку уведомлений исполнителям.

Такой вариант не является полноценной микросервисной архитектурой с независимыми базами данных. Для учебного задания это осознанное упрощение: система небольшая, поэтому достаточно разделить ответственность по контейнерам, но не усложнять модель лишней инфраструктурой.

## Роли пользователей

### Руководитель / автор цели

Создает цели, добавляет задачи на пути к цели, назначает исполнителей, просматривает задачи цели и контролирует их статус.

### Исполнитель

Просматривает назначенные задачи, открывает список задач по цели и изменяет статус своей задачи.

### Администратор

Создает пользователей и выполняет поиск пользователей по логину, имени и фамилии.

## Внешние системы

### Email/SMS Notification Provider

Внешний сервис для отправки уведомлений. Используется, когда исполнителю назначена новая задача или изменен статус задачи.

Если преподаватель требует совсем минимальную модель без внешних систем, этот элемент можно удалить. Но для сервиса планирования задач уведомления выглядят логично и помогают показать интеграцию с внешней системой.

## Основные use cases

| Роль | Use case | Реализация |
|---|---|---|
| Администратор | Создать пользователя | Web App → API Gateway → User Service → PostgreSQL |
| Администратор | Найти пользователя по логину | Web App → API Gateway → User Service → PostgreSQL |
| Администратор / руководитель | Найти пользователя по имени и фамилии | Web App → API Gateway → User Service → PostgreSQL |
| Руководитель | Создать новую цель | Web App → API Gateway → Goal Service → PostgreSQL |
| Руководитель / исполнитель | Получить список всех целей | Web App → API Gateway → Goal Service → PostgreSQL |
| Руководитель | Создать задачу на пути к цели | Web App → API Gateway → Task Service → Goal Service / User Service → PostgreSQL → Notification Service |
| Руководитель / исполнитель | Получить все задачи цели | Web App → API Gateway → Task Service → PostgreSQL |
| Исполнитель | Изменить статус задачи | Web App → API Gateway → Task Service → PostgreSQL → Notification Service |

## Контейнеры системы

### Web Application

Клиентское веб-приложение для работы с целями, задачами и пользователями. Отправляет запросы в Backend API по HTTPS/REST.

### API Gateway

Единая точка входа для клиентского приложения. Выполняет маршрутизацию запросов к доменным сервисам.

### User Service

Отвечает за создание пользователей и поиск пользователей по логину, имени и фамилии.

### Goal Service

Отвечает за создание целей и получение списка целей.

### Task Service

Отвечает за создание задач внутри цели, получение задач цели и изменение статуса задачи.

### Notification Service

Отвечает за подготовку и отправку уведомлений через внешний Email/SMS provider.

### PostgreSQL Database

Хранит основные данные: пользователей, цели, задачи, исполнителей и историю изменения статусов.

## Архитектурно значимый сценарий для dynamic diagram

Для dynamic diagram выбран сценарий **создания новой задачи на пути к цели**. Он важен архитектурно, потому что затрагивает сразу несколько частей системы:

1. пользовательский интерфейс;
2. API Gateway;
3. Task Service;
4. Goal Service, так как задача должна относиться к существующей цели;
5. User Service, так как у задачи должен быть существующий исполнитель;
6. PostgreSQL, где сохраняется задача;
7. Notification Service, который уведомляет исполнителя.

## Как открыть диаграммы в Structurizr

Для просмотра диаграмм нужен установленный Docker Desktop.

1. Открыть терминал и перейти в папку с файлами домашнего задания:

```bash
cd путь_к_папке/homework_01
```

2. Проверить, что в папке есть файлы:

```bash
ls -la
```

В папке должны находиться:

- `readme.md`
- `workspace.dsl`

3. Запустить Structurizr локально через Docker:

```bash
docker run -it --rm -p 8080:8080 -v "$(pwd)":/usr/local/structurizr structurizr/structurizr local
```

4. Открыть в браузере:

```text
http://localhost:8080
```

После открытия будут доступны диаграммы:

- System Context;
- Container;
- Dynamic diagram для сценария создания задачи на пути к цели.

Важно: пока диаграммы открыты в браузере, терминал с запущенным Docker-контейнером закрывать не нужно.

Если порт `8080` занят, можно запустить на другом порту:

```bash
docker run -it --rm -p 8081:8080 -v "$(pwd)":/usr/local/structurizr structurizr/structurizr local
```

И открыть:

```text
http://localhost:8081
```
